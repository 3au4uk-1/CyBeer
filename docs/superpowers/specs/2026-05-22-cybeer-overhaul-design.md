# CyBeer Overhaul — дизайн-спецификация

Дата: 2026-05-22

## Обзор

Комплексное обновление CyBeer: исправление проблем прошивки, переработка веб-интерфейса, добавление новых фич. Реализация в два прохода: сначала firmware, затем frontend.

### Скоуп

**Проход 1 — Firmware fixes + backend:**
- Firmware version из сборки
- NTP-синхронизация
- Расширение WebSocket лимита
- Оптимизация LED-рендера
- Фильтрация leaderboard
- Новые API-эндпоинты
- Увеличение подиум-эффекта

**Проход 2 — Frontend overhaul:**
- Унификация навигации и языка (русский)
- Переделка главной (лидерборд)
- Переделка claim, player, admin, tournament
- Spectator / TV mode
- Звук при финише

### Исключено из скоупа

- Автоперезапуск в PREP по таймауту (отложено до теста в реальных условиях)
- PWA / service worker
- Аватарки / эмодзи участников
- Командный режим
- Achievement-система
- QR-код
- Пьезо-зуммер (звук только в браузерах)
- i18n-инфраструктура (язык один — русский)

## Проход 1: Firmware fixes + backend

### 1.1 Firmware version из сборки

**Проблема:** `"1.0.0"` захардкожена в `cybeer_web.c` (`h_get_status`).

**Решение:** Заменить строку на макрос `PROJECT_VER`, который ESP-IDF автоматически определяет из `project(... VERSION x.y.z)` в корневом `CMakeLists.txt`.

**Файлы:**
- `firmware/CMakeLists.txt` — убедиться что версия задана в `project()`
- `firmware/components/cybeer_web/cybeer_web.c` — замена `"1.0.0"` на `PROJECT_VER`

### 1.2 NTP-синхронизация

**Проблема:** `finished_at` содержит бессмысленные epoch-placeholder'ы при отсутствии синхронизации времени.

**Решение:** При получении IP в STA-режиме запускаем SNTP-клиент с сервером `pool.ntp.org`. Используем `esp_sntp` API из ESP-IDF.

**Поведение:**
- Вызов `esp_sntp_init()` в колбэке `IP_EVENT_STA_GOT_IP` в `cybeer_wifi.c`
- Один сервер: `pool.ntp.org`
- Синхронизация выполняется в фоне, без блокировки
- Если синхронизация не произошла — поведение как сейчас (epoch placeholder)
- Повторных настроек не требуется, никакого UI

**Файлы:**
- `firmware/components/cybeer_wifi/cybeer_wifi.c` — инициализация SNTP при GOT_IP

### 1.3 WebSocket лимит: 4 → 10

**Проблема:** `CYBEER_WS_MAX_CLIENTS = 4` — на вечеринке с 10+ гостей слоты заканчиваются, остальные без live-обновлений.

**Решение:** Увеличить до 10. Каждый слот — один `int` fd + периодическая отправка коротких JSON. На ESP32-C3 с ~400KB RAM это допустимо.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_ws.c` — `CYBEER_WS_MAX_CLIENTS` 4 → 10

### 1.4 LED: убрать чтение storage из горячего цикла

**Проблема:** В `render_frame()` после FINISHED flash+dim (880ms) каждые ~20ms вызывается `cybeer_storage_get_latest_unclaimed_run_id()` — парсинг JSON с файловой системы в hot loop.

**Решение:** Кэшированный флаг `s_has_unclaimed_run` (bool) в RAM. Обновляется:
- `true` — при сохранении нового рана (`on_finished_placeholder`)
- `false` — при claim (`h_post_claim` после успешного `cybeer_storage_claim_run`)

В `render_frame()` проверяем только флаг, без обращения к storage.

**Интерфейс:**
- Новая функция `cybeer_led_set_unclaimed_flag(bool has_unclaimed)` в `cybeer_led.h`
- Вызывается из `app_main.c` (on_finished) и `cybeer_web.c` (claim handler, data reset handler)

**Важно:** при `DELETE /api/admin/data/reset` тоже нужно вызвать `cybeer_led_set_unclaimed_flag(false)`, чтобы LED не мигал маяком на несуществующий ран.

**Файлы:**
- `firmware/components/cybeer_led/cybeer_led.c` — флаг + замена storage-вызова
- `firmware/components/cybeer_led/include/cybeer_led.h` — новая функция
- `firmware/main/app_main.c` — вызов при сохранении рана
- `firmware/components/cybeer_web/cybeer_web.c` — вызов при claim и data reset

### 1.5 Leaderboard: фильтрация duration_us ≤ 0

**Проблема:** Мануально добавленные заезды с `duration_us = 0` попадают на первое место лидерборда.

**Решение:** В `h_get_leaderboard` при итерации по claimed ранам пропускаем записи с `duration_us <= 0`.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_web.c` — `h_get_leaderboard`

### 1.6 Новый API: заезды участника

**Эндпоинт:** `GET /api/participants/:id/runs`

**Ответ:** JSON-массив ранов с `participant_id == :id`, отсортированных по `finished_at` (новые первые). Лимит — последние 50.

**Реализация:** Фильтрация из `runs.json` по `participant_id`, аналогично `h_get_runs`, но с фильтром.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_web.c` — новый handler + регистрация URI
- `firmware/components/cybeer_storage/include/cybeer_storage.h` — (опционально) вспомогательная функция

### 1.7 Эндпоинт проверки PIN

**Эндпоинт:** `POST /api/admin/pin/verify`

**Заголовок:** `X-Admin-Pin`

**Ответ:**
- `200 {"ok":true}` — PIN верный
- `401 {"error":"invalid pin"}` — PIN неверный
- `403 {"error":"admin pin not configured"}` — PIN не настроен

**Реализация:** Переиспользует `require_admin_pin()`, но при успехе возвращает `{"ok":true}` вместо продолжения обработки.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_web.c` — новый handler + регистрация URI

### 1.8 Новый API: список турниров

**Эндпоинт:** `GET /api/admin/tournaments`

**Заголовок:** `X-Admin-Pin`

**Ответ:** JSON-массив всех турниров (из `tournaments.json`). Нужен для dropdown'а в турнирной секции админки (выбор турнира для старта).

**Реализация:** Переиспользует `cybeer_storage_tournaments_json()` (функция уже есть, но не выставлена через HTTP). Оборачиваем в handler с `require_admin_pin()`.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_web.c` — новый handler + регистрация URI

### 1.9 Подиум: 3с → 6с

**Проблема:** Подиум-эффект на LED-ленте длится 3 секунды — слишком коротко для празднования.

**Решение:** `s_podium_until_us = now + 6000000` (6 секунд).

**Файлы:**
- `firmware/components/cybeer_led/cybeer_led.c` — константа

### 1.10 unclaimedRunDuration в статусе

**Проблема:** `GET /api/status` возвращает `unclaimedRunId`, но не время заезда. Фронтенду нужно показывать время при claim.

**Решение:** Добавляем поле `unclaimedRunDurationUs` (число, микросекунды) в ответ `/api/status`. Если нет незаклеймленного рана — поле отсутствует или `null`.

**Файлы:**
- `firmware/components/cybeer_web/cybeer_web.c` — `h_get_status`

## Проход 2: Frontend overhaul

### 2.1 Общая структура и навигация

**Все страницы** (кроме `setup.html` и `spectator.html`) получают единый топбар:

```
[🍺 CyBeer]  [батарея] [состояние] [fw ver]
[Главная] [Заявить заезд] [Турнир] [Админка]
```

- Топбар выносится в переиспользуемый фрагмент в `app.js` (динамическая генерация, т.к. vanilla JS без шаблонизатора)
- Язык всего UI — русский
- `setup.html` сохраняет свои inline-стили (captive portal на AP)
- `spectator.html` — без навигации (fullscreen для ТВ)

**Файлы:** все HTML-файлы + `app.js` + `app.css`

### 2.2 Главная страница (index.html)

**Структура:**

1. **Live-таймер** — крупный блок сверху:
   - PREP: "Готов к старту" (зелёный акцент)
   - RUNNING: большие цифры времени (оранжевый пульс)
   - FINISHED: результат + "ожидает заявки" (золотой)
   - READY: последний результат (приглушённо)

2. **Лидерборд** — таблица снизу:
   - Данные из `GET /api/leaderboard`
   - Колонки: #, Имя, Время, Дата
   - Имя — ссылка на `player.html?id=...`
   - Обновляется по WS-событию `leaderboardUpdate`

**Убираем:** таблицу "Recent runs" (информация для админки, не для зрителей).

**Данные:** WS для timer/state/runFinished/leaderboardUpdate, REST для начальной загрузки лидерборда.

### 2.3 Claim (claim.html)

**Изменения относительно текущего:**
- Топбар с навигацией
- Русский язык
- Показываем время незаклеймленного заезда (из `unclaimedRunDurationUs` статуса), а не только UUID
- UUID скрыт или показан мелко — человек видит "Незаявленный заезд: **4.32с**"
- После успешного claim: "Заявлено! Ваш результат: 4.32s"
- WS-подключение для live-обновления статуса (появление нового рана)

### 2.4 Player (player.html)

**Полная переделка.** URL: `/player.html?id=<uuid>`

**Структура:**
1. **Карточка участника:** имя крупно, позиция в лидерборде
2. **Статистика:** рекорд, среднее, худшее, количество заездов (из `GET /api/participants/:id/stats`)
3. **Таблица заездов:** все заезды участника (из `GET /api/participants/:id/runs`) — дата, время

**Стилистика:** общий `app.css`, топбар с навигацией.

### 2.5 Admin (admin.html)

**Доработки:**

1. **PIN-валидация при вводе:** при нажатии "Запомнить PIN" — запрос на `POST /api/admin/pin/verify`. Если неверный — ошибка сразу, админка не открывается.

2. **LED: текущие значения из API:** при загрузке страницы подтягиваем `ledCount` и `ledBrightness` из `GET /api/status` и подставляем в форму.

3. **Турнирная секция** (новый блок в админке):
   - **Создание:** текстовое поле "Название турнира" + чекбоксы участников (из `GET /api/participants`). Кнопка "Создать" → `POST /api/admin/tournaments`.
   - **Старт:** dropdown со списком созданных турниров + кнопка "Старт" → `POST /api/admin/tournaments/:id/start`.
   - **Назначение слотов:** dropdown матчей текущего турнира, radio A/B, кнопка "Привязать следующий заезд" → `POST /api/admin/tournaments/matches/:id/assign`.

4. **Русский язык:** все надписи, placeholder'ы, сообщения.

### 2.6 Tournament (tournament.html)

**Улучшения визуализации:**
- Текущий активный матч (из `pendingNextRun`) — жёлтая рамка + пульсация CSS
- Завершённый матч с победителем — зелёная рамка, имя победителя выделено
- Чемпион (если определён) — золотая карточка наверху страницы, крупно
- Русский язык

**Без изменений:** структура данных, polling 8с, общий layout сетки.

### 2.7 Spectator / TV mode (spectator.html)

**Новая страница.** Оптимизирована для ТВ/монитора (landscape, fullscreen).

**Переключение режимов:** URL-параметр `?mode=leaderboard` (по умолчанию) / `?mode=tournament`. На экране — минималистичная иконка для переключения (единственный интерактивный элемент, кроме кнопки звука).

**Режим "leaderboard":**
- Верхняя половина: live-таймер крупными цифрами
  - PREP — зелёный акцент
  - RUNNING — оранжевая пульсация
  - FINISHED — вспышка белого → золотой результат
- Нижняя половина: топ-10 лидерборд крупным шрифтом
  - При обновлении — CSS-анимация подсветки новой/изменённой строки

**Режим "tournament":**
- Верхняя половина: тот же live-таймер
- Нижняя половина: турнирная сетка (упрощённая, крупная)
  - Текущий матч выделен
  - Анимация при определении победителя

**Технические детали:**
- Только WebSocket для обновлений (без polling)
- `cursor: none` через CSS после 3с без движения мыши
- Тёмный фон (пивная тема), высокий контраст
- Нет скролла — всё помещается на экран
- Нет навигации, нет батареи — чистый экран для зрителей

### 2.8 Звук при финише

**Триггер:** WS-событие `runFinished`.

**Реализация:** синтез фанфары через Web Audio API (2-3 секунды, мажорная последовательность из нескольких нот). Без внешнего файла — генерируется в коде.

**Где работает:** все страницы с WS-подключением — `index.html`, `spectator.html`, `claim.html`.

**Browser autoplay policy:** браузеры блокируют звук до первого user interaction. Решение:
- На `spectator.html`: кнопка "Включить звук" при первом открытии. После клика AudioContext разблокирован.
- На остальных страницах: звук активируется после любого клика/тапа (claim, навигация). Если пользователь не взаимодействовал — звук не играет (это ок, они не основная аудитория для звука).

**Звук:** синтезированная фанфара — 3 восходящие ноты (C5, E5, G5) с затуханием, sine wave. Простая, но праздничная. Код генерации — отдельная функция в `app.js`.
